Trails v1.5 ships three improvements that make cross-chain intents cheaper, more flexible, and more composable:

1. **PERMIT flow** — EIP-2612 tokens skip the separate approval transaction entirely.
2. **HydrateProxy** — Destination calldata is committed malleable: exact amounts are injected at execution time, not quote time.
3. **Composable Actions** — The `actions: []` array in `useQuote` and `useTrailsSendTransaction` is the high-level interface over HydrateProxy.

v1.5 is the default as of the current SDK. You do not need to set `intentProtocol` unless you have a specific reason to downgrade to v1.

---

## PERMIT flow

### The problem with v1

For ERC-20 tokens, an intent requires the user's funds to move from their wallet to an intent address. In v1 this required two user signatures on the origin chain:

1. An ERC-20 `approve` transaction (paying gas).
2. The intent signature.

### How v1.5 handles it

For tokens that implement [EIP-2612](https://eips.ethereum.org/EIPS/eip-2612) (USDC, DAI, and most modern ERC-20s), the SDK requests a single gasless `permit` signature instead of an on-chain approval. The permit is bundled with the intent and submitted together — the user signs once, and the token approval + intent initiation happen atomically.

This is what the "1-click" claim means for EIP-2612 tokens: no approval transaction, no waiting for it to land, no second confirmation prompt.

```
v1:   approve tx (gas) → sign intent → done (2 interactions)
v1.5: sign permit + intent together → done (1 interaction)
```

For tokens that do not support EIP-2612, the flow falls back to a standard approval before the intent signature. No configuration is required — the SDK detects permit support automatically.

### What this requires from you

Nothing. The SDK handles permit detection and signing internally. If you are passing an ERC-20 token on the origin chain, v1.5 will automatically attempt a permit signature before falling back to an on-chain approval.

## HydrateProxy and malleable calldata

### The problem with static calldata commitment

In v1, when your intent included destination calldata (a contract call on the destination chain), the exact amount had to be known at quote time and locked into the signed payload. If the bridged amount differed from the quoted amount due to slippage or bridge fees, the destination call would either fail or leave dust.

### How HydrateProxy solves it

v1.5 introduces `HydrateProxy`, a contract module that receives a compact description of what values to inject, reads those values from the chain at execution time, and substitutes them into the calldata before forwarding the call.

Instead of committing `deposit(1000000)` at quote time, you commit a hydration instruction: "when executing, read the ERC-20 balance of the intent wallet and substitute it at offset N in the calldata." The actual number is determined at the moment of execution — whatever arrived is what gets deposited.

This is what makes `dynamic()` work in Composable Actions:

```ts
import { lend, dynamic } from '0xtrails'

// Lend whatever USDC actually arrives — no need to predict bridge output.
const actions = [
  lend({
    marketId: 'base-usdc-aave-v3-lending',
    amount: dynamic(),
  }),
]
```

### Malleable vs static calldata

| | Static (v1) | Malleable (v1.5) |
|---|---|---|
| Amount in signed payload | Fixed at quote time | Placeholder; hydrated at execution time |
| Slippage risk | Destination call can fail if amount differs | Execution uses actual on-chain balance |
| Use case | Known exact amounts | `dynamic()`, `self()`, multi-step chaining |
| Commitment security | Full calldata committed | Hydration rules committed; values resolved on-chain |

HydrateProxy enforces that only the pre-committed hydration rules can modify the calldata. The relayer cannot inject arbitrary values — the substitution logic is locked in the signed payload and verified on-chain.

## Composable Actions

Composable Actions are the high-level interface over HydrateProxy. Instead of encoding calldata and hydration payloads manually, you express destination execution as a typed list of steps.

### Passing actions to `useQuote`

```tsx
import { useQuote, dynamic, swap, lend } from '0xtrails'

const { send, isLoadingQuote } = useQuote({
  from: { chain: 'arbitrum', token: 'USDC' },
  to:   { chain: 'base',     token: 'USDC', amount: '100' },
  actions: [
    swap({
      tokenIn: 'USDC',
      tokenOut: 'WETH',
      amountIn: dynamic(),
    }),
    lend({
      marketId: 'base-weth-aave-v3-lending',
      amount: dynamic(),
    }),
  ],
})
```

### Passing actions to `useTrailsSendTransaction`

```tsx
import { useTrailsSendTransaction, dynamic, deposit } from '0xtrails'

const { sendTransaction, isPending } = useTrailsSendTransaction({
  actions: [
    deposit({
      marketId: 'base-usdc-morpho-0x...-4626-vault',
      amount: dynamic(),
    }),
  ],
})

// Trigger the Trails modal — user picks origin token and chain.
sendTransaction({
  to: recipientAddress,
  tokenAddress: '0x833589fcd6edb6e08f4c7c32d4f71b54bda02913', // USDC on Base
  tokenAmount: '100000000', // 100 USDC (6 decimals)
})
```

Both hooks convert the `actions` array into hydrated calldata using v1.5 encoding before submitting the quote.

See the full [Composable Actions documentation](/sdk/composable-actions/overview) for available builders, dynamic values, and market discovery.

## `intentProtocol` option

`intentProtocol` is an optional override on `TrailsProvider` and `useQuote`. The default is `"v1.5"`.

```tsx
import { TrailsProvider } from '0xtrails'

// Default — no configuration needed.
<TrailsProvider config={{ trailsApiKey: "..." }}>
  {children}
</TrailsProvider>

// Explicit override (not recommended unless you have a specific reason).
<TrailsProvider
  config={{
    trailsApiKey: "...",
    intentProtocol: "v1",
  }}
>
  {children}
</TrailsProvider>
```

You can also pass `intentProtocolVersion` per-quote in `useQuote` if you need to mix protocols within the same app, though this is an edge case.

| Setting | Behavior |
|---|---|
| `"v1.5"` (default) | PERMIT flow, HydrateProxy for dynamic calldata, Composable Actions support |
| `"v1"` | Legacy behavior: on-chain approval + infinite permit to entrypoint, static calldata |
